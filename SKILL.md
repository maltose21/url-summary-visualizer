# URL Summary Visualizer — Agent Team Edition

## 团队架构

```
                    ┌─────────────────┐
                    │     主编        │
                    │  Team Lead     │
                    │  - 分配任务     │
                    │  - 决策布局    │
                    │  - 审核把关    │
                    │  - 最终交付    │
                    └────────┬────────┘
                             │ 任务分配 & 审核
        ┌────────────────────┼────────────────────┐
        ▼                    ▼                    ▼
┌───────────────┐   ┌───────────────┐   ┌───────────────┐
│  内容分析师    │   │  图表设计师    │   │  布局设计师    │
│ Content       │   │  Diagram      │   │  Layout       │
```

## 角色边界（严格划分）

### 主编 (Team Lead)

| 职责 | 说明 |
|------|------|
| **任务分配** | 决定何时分配什么任务给哪个队友 |
| **布局决策** | 根据内容分析师的建议，选择最合适的布局模板 |
| **审核把关** | 检查产出是否符合标准，不达标则打回 |
| **流程控制** | 决定流水线何时进入下一阶段 |
| **最终交付** | 确认达标后，告知用户文件路径 |

**主编不做：**
- 不亲自抓取内容（交给内容分析师）
- 不亲自设计 SVG（交给图表设计师）
- 不亲自写 HTML（交给布局设计师）

### 内容分析师 (Content Analyst)

**唯一职责：忠实提取，不做决策。**

输入：URL
输出：结构化分析报告

```
## 内容分析报告

**页面主题**: [从页面标题/TOPIC提取]
**内容来源**: [URL]
**内容类型**: [文档 / 教程 / 工具介绍 / 新闻 / 数据 / 混合]
**结构特征**: [步骤流 / 层次结构 / 并列要点 / 架构关系 / 时间线 / 混合]

**核心要点**（3-7条，忠实于原文）:
1. [要点1标题]：[支撑这句话的原文内容]
2. [要点2标题]：[支撑这句话的原文内容]
3. ...

**关键数据**（如有数字/指标）:
- [指标名]: [数值] — [来源段落]

**布局建议**（仅建议，决策权在主编）:
- 推荐布局: [Hero / Flowchart / Stats / Steps / Timeline]
- 理由: [为什么这个布局适合当前内容]
- 图表需求: [架构图 / 流程图 / 时间线 / 对比表 / 无 / 待定]
```

**执行：**
```bash
${BUN_X} /Users/cuizhaoming1/.claude/skills/baoyu-url-to-markdown/scripts/main.ts <URL> -o /tmp/url-content.md
```
读取 `/tmp/url-content.md`，按上述格式输出。**不添加个人解读，不编造内容。**

### 图表设计师 (Diagram Designer)

**唯一职责：基于既定内容，绘制准确、简洁、美观的 SVG 图表。**

输入：内容分析报告（来自主编转发）
输出：SVG 代码（纯文本，可直接嵌入 HTML）

**不决定布局，不决定配色（遵循预设规范）。**

#### SVG 绘制规范（必须严格遵守）

```svg
配色变量（预设，不可自行更改）:
  accent-soft: #FFF4F0   ← 强调区块背景
  accent:      #FF6B35   ← 强调文字/边框
  white:       #FFFFFF   ← 普通区块
  gray-light:  #F5F5F7   ← 次要区块
  gray-border: #D2D2D7   ← 边框/分割线
  text-primary: #1D1D1F  ← 主要文字
  text-secondary: #6E6E73 ← 次要文字
  text-tertiary: #86868B ← 辅助文字

字体: font-family="Inter,-apple-system,BlinkMacSystemFont,sans-serif"
圆角: rx="8" ~ rx="12"
线宽: stroke-width="1" ~ "2"
```

#### 根据内容类型选择图表

| 内容结构 | SVG 类型 | 说明 |
|---------|---------|------|
| 系统架构/组件关系 | 分区卡片 + 连接线 | 用 `<rect>` 画区块，`<line>`+`<marker>` 画连线 |
| 流程/步骤 | 横向步骤流 | 圆角矩形 + 箭头连接 |
| 层次/包含 | 树状图 | 根节点在上，子节点在下，逐层展开 |
| 时间线/演进 | 横向时间线 | 横线 + 圆点 + 标签 |
| 并列对比 | **不用画图** | 直接用 HTML `<table>`，图表设计师不需要介入 |

#### SVG 输出格式

```svg
<!-- 直接输出此格式，主编会嵌入 HTML -->
<svg viewBox="0 0 {W} {H}" fill="none" xmlns="http://www.w3.org/2000/svg">
  <defs>
    <marker id="arr" markerWidth="8" markerHeight="8" refX="6" refY="3" orient="auto">
      <path d="M0,0 L0,6 L8,3 z" fill="#D2D2D7"/>
    </marker>
  </defs>
  <!-- 你的图表代码 -->
</svg>
```

**图表设计师产出：** SVG 代码文本块（纯文本），标注 `viewBox` 宽高。

### 布局设计师 (Layout Designer)

**唯一职责：将分析报告 + SVG 方案转化为完整的单屏 HTML 文件。**

输入：
- 内容分析报告（主编转发）
- SVG 代码（图表设计师产出）

输出：`/tmp/url-summary.html`

**执行步骤：**

1. **确认布局**（主编已决策，布局设计师照做）
2. **组装 HTML**：按模板填入内容
3. **嵌入 SVG**：将图表设计师的 SVG 代码放入 `.diagram-wrap` 内
4. **质量自检**（layout designer 自己检查，非主编职责）：
   - [ ] 单屏不溢出（`overflow: hidden`，100vh）
   - [ ] SVG 标签全部闭合
   - [ ] 无硬编码内容（所有文字来自分析报告）
   - [ ] Badge 与内容主题匹配
   - [ ] 日期为当天

**HTML 规范（必须遵守）：**

```html
<!-- 单屏容器 -->
<body style="height: 100vh; overflow: hidden; ... ">
  <div class="page" style="height: calc(100vh - 48px); ...">
    <!-- 内容 -->
  </div>
</body>

<!-- Google Fonts（唯一外部依赖，若失败降级系统字体）-->
<link href="https://fonts.googleapis.com/css2?family=Inter:wght@400;500;600;700&display=swap" rel="stylesheet">

<!-- 无 Mermaid，无 Chart.js，无其他 CDN -->
```

## 驳回与修正机制

### 驳回触发条件

以下情况出现时，主编必须驳回：

**Step 1（内容分析）被驳回的条件：**
- 要点数量不在 3-7 条范围内
- 要点内容在原文中无对应支撑
- 出现主观解读、延伸推断或编造内容
- 内容类型判断与实际明显不符
- 遗漏了页面中的重要章节或数据

**Step 2（图表设计）被驳回的条件：**
- SVG 标签未闭合，存在语法错误
- 图表节点数量与实际要点数量不匹配
- 连接关系与内容逻辑不符（如箭头指向错误）
- 图表元素严重重叠，视觉效果混乱
- 未使用预设配色规范

**Step 4（HTML 生成）被驳回的条件：**
- 出现滚动条，100vh 溢出
- 存在硬编码内容（文字与分析报告不符）
- Badge 与内容主题无关或明显错误
- Footer 日期不是当天
- 未嵌入 SVG 或 SVG 渲染异常

### 修正与重新提交流程

**被驳回者的响应规范：**

1. **读取驳回报告**：收到主编的《驳回报告》后，先完整阅读
2. **确认驳回编号**：在修正报告中注明 `回应 R-001`
3. **逐条修复**：针对驳回报告中的每一条问题，分别修改
4. **标注改动**：修正后用 `<!-- R-001 修改点: ... -->` 注释标注关键改动
5. **提交修正报告**：连同修正后的产出一起提交

### 驳回次数上限与升级机制

| 次数 | 处理方式 |
|------|---------|
| 第 1 次驳回 | 正常修正流程，被驳回者修正后重新提交 Step |
| 第 2 次驳回 | 修正后重新提交，并附上修正报告，说明仍未解决问题的原因 |
| 第 3 次驳回 | **主编召集全体会议**：被驳回者说明困难，主编明确修改方向后，被驳回者必须按照主编明确的指令修改，不得再自行判断 |
| **超过 3 次** | 主编有权更换执行者：指定其他角色完成，或亲自执行该环节，但更换执行者后仍算作一次驳回（仍受上限约束） |

**主编有权终止流水线的情况：**
- 同一环节被驳回超过 5 次（含）
- 主编判断被驳回者的能力无法完成当前任务
- URL 本身无法抓取或内容为空

终止时主编必须向用户说明原因。

## 主编审核清单

**执行者：主编（唯一审核者）**

| 检查项 | 标准 | 不通过时打回 |
|--------|------|------------|
| **内容准确性** | 所有文字来自源内容，无编造、无幻觉 | Step 1 |
| **要点完整性** | 核心要点覆盖页面主要内容，要点数量 3-7 条 | Step 1 |
| **图表准确性** | SVG 节点数量、连接关系与内容一致；标签闭合 | Step 2 |
| **图表美观** | 节点间距合理、箭头指向正确、无元素重叠 | Step 2 |
| **布局匹配** | 选用布局与内容结构匹配（主编的决策验证） | Step 3 |
| **单屏不溢出** | 100vh 内无滚动条 | Step 4 |
| **视觉规范** | Anthropic 配色、圆角卡片、阴影、hover 动画 | Step 4 |
| **Badge 合理** | Badge 文字反映内容主题（如"文档"、"工具"、"教程"） | Step 4 |
| **日期正确** | Footer 日期为当天 | Step 4 |

**审核结论格式：**

```
【主编审核 R-XXX】
审核时间：[年-月-日 HH:MM]

□ 全部通过 → 结论：通过
□ 部分通过（列出未通过项）→ 结论：驳回，输出《驳回报告》R-XXX

未通过项：
1. [检查项] — [具体问题]
2. ...
```

### 内容分析师的自进化

**进化目标：对同类内容的分析越来越精准。**

**记忆写入（Step 1 完成后）：**
```jsonl
// ~/.url-summary-visualizer/memory/analyst/past-analyses.jsonl
{"task_id":"xxx","url":"...","content_type":"文档","structure":"层次结构","points":[{"title":"...","source_paragraph":"..."}],"layout_suggested":"Hero","rejection_count":0,"score":5,"timestamp":"..."}
```

**进化机制：**
- 每次分析新 URL 前，先读取同类内容的分析记录作为参考
- 内容类型判断准确率 → 记录在 `expertise.json`
- 如果某类内容的要点提取被驳回，该类型会积累"高风险信号"，下次分析时更仔细

**分析师查记忆的时机：**
> "分析此 URL 前，先检查记忆中是否有相同/相似内容类型的历史记录，参考其中的要点提取模式。"

```
分析师进化指标（来自 expertise.json）：
- 文档类准确率：95%（基于20次任务）
- 工具介绍类准确率：88%（基于12次任务）
- 高风险：数据类内容（易遗漏数字细节）
```

### 图表设计师 + 布局设计师的自进化

**进化目标：页面越来越美观、协调，不断汲取设计理念。**

**记忆写入（每次交付后）：**

**被驳回 Pattern 积累：**
```jsonl
// ~/.url-summary-visualizer/memory/designer/rejected-patterns.jsonl
{"pattern":"节点数量>6时未调整间距","count":3,"last_occurred":"...","fix_applied":"增加viewBox宽度或改用更小字体"}
```

**风格库积累：**
```jsonl
// ~/.url-summary-visualizer/memory/designer/style-repertoire.jsonl
```

- 参考 `style-repertoire.json` 中同布局的高分方案
- 用户对"美观"的打分直接影响 style-repertoire 的权重

**设计师查记忆的时机：**
> "在开始 SVG 设计前，先检查被驳回 Pattern 列表和成功风格库。"


**进化目标：审核越来越精准，驳回越来越少，误判率持续下降。**

### 自进化系统的使用规则

1. **每次任务前**：各角色必须先读取自身记忆库，再开始工作
2. **每次任务后**：主编触发记忆写入，不得遗漏
3. **记忆优先级**：越近期的记忆权重越高（近期 10 次 > 早期历史）
4. **记忆质量**：被打回修正的记忆记录，标注 `rejected: true`，影响后续参考权重
5. **你的打分**：你的评分是记忆系统最重要的输入，直接驱动三个角色的进化方向

### 记忆库的初始化

如果是首次使用（记忆库为空），各角色以预设规范为基准，按本次 skill 文档执行，不参考历史。
