# URL Summary Visualizer — Agent Team Edition

融合「横纵分析法」的深度 URL 可视化工具。

## 方法论核心：横纵分析法

**纵轴（纵向分析）**：沿时间维度，追踪研究对象从诞生到当下的完整生命历程。
- 起源与背景、诞生节点、关键演进、决策逻辑、阶段划分
- 叙事驱动，还原发展全貌

**横轴（横向分析）**：在当下时间截面，与竞品/同类进行系统性对比。
- 功能对比、定位差异、适用场景、优劣势分析
- 不止于罗列，在于发现竞争格局中的独特位置

**交汇（洞察）**：将历史脉络与竞争格局结合。
- 回答"历史如何塑造当下位置"
- 给出独到洞察和未来推演

## 角色边界（严格划分）

### 主编 (Team Lead)

| 职责 | 说明 |
|------|------|
| **任务分配** | 决定何时分配什么任务给哪个队友 |

### 内容分析师 (Content Analyst)

**唯一职责：从 URL 抓取内容，完成初步横纵分析框架，不做结论。**

输入：URL
输出：结构化分析框架（纵向分析 + 横向分析 + 待验证假设）

**执行步骤：**

1. **抓取内容**：使用 baoyu-url-to-markdown 将 URL 转为 markdown
2. **纵向分析**：起源、背景、诞生节点、演进历程、关键决策
3. **横向分析**：同类竞品、核心差异、定位差异、优劣势框架
4. **识别盲区**：明确标注"需要外部验证"的部分（历史细节、竞品数据、用户反馈）

### 外部信息检索员 (External Researcher)

**唯一职责：接收内容分析师的盲区标记，搜索外部材料补充验证。**

输入：内容分析框架（含盲区标记）
输出：外部信息包（第三方来源汇总）

**搜索策略：**

1. **技术博客**：搜索"[产品名] 技术深度解析"、"[产品名] 架构分析"
2. **用户反馈**：搜索"[产品名] 评价"、"[产品名] vs [竞品] 体验对比"
3. **社区讨论**：搜索"[产品名] Reddit/HN 讨论"、"[产品名] 优缺点"
4. **竞品对比**：搜索"[竞品A] vs [产品名]"、"[产品名] 对比 [竞品B]"
5. **数据来源**：搜索"[产品名] 用户数"、"[产品名] 市场份额"

### 执笔编辑 (Lead Writer)

**整合原则：**

1. **优先外部证据**：与原文档冲突时，以第三方来源为准

### 图表设计师 (Diagram Designer)

**唯一职责：基于最终分析报告，绘制准确、简洁、美观的 SVG 图表。**

输入：最终分析报告（来自执笔编辑）
输出：SVG 代码（纯文本，可直接嵌入 HTML）

**不决定布局，不决定配色（遵循预设规范）。**

---

#### 一、图表类型选择指南

根据内容本质选择图表类型，不要机械套用：

| 内容特征 | 推荐图表 | 原因 |
|---------|---------|------|
| 时间演进、阶段推进 | `timeline`（横向时间线） | 沿X轴展示历史阶段，圆点标记关键节点 |
| 市场定位、策略选择 | `matrix_2x2`（四象限） | 双维度交叉找到独特位置 |
| 系统架构、组件关系 | `org_chart` / 卡片+连接线 | 节点+连线表达层次和依赖 |
| 流程步骤、方法论 | `chevron_process` / `process_flow` | 箭头链式展示顺序 |
| 多维对比、竞品分析 | `comparison_table`（HTML table） | 结构化数据用表格更清晰 |
| 优势劣势、能力评估 | `radar_chart`（雷达图） | 多维度并排可视化 |
| 构成比例、份额 | `donut_chart`（甜甜圈图） | 中心可放 KPI，环形展示比例 |
| 趋势变化、增长 | `line_chart` / `area_chart` | 连续数据展示变化 |
| 排名对比 | `horizontal_bar_chart`（横向条形） | 长标签更适合横向布局 |

**选错图表类型是最常见的错误。** 拿到内容后先问：这个内容想传达的核心信息是什么？是时间推移、空间定位、功能对比、还是构成比例？答案决定图表类型。

#### 二、SVG 构造规范

**配色变量（预设，不可自行更改）：**

```
accent-soft: #FFF4F0   ← 强调区块背景
accent:      #FF6B35   ← 强调文字/边框
white:       #FFFFFF   ← 普通区块
gray-light:  #F5F5F7   ← 次要区块
gray-border: #D2D2D7   ← 边框/分割线
text-primary: #1D1D1F  ← 主要文字
text-secondary: #6E6E73 ← 次要文字
text-tertiary: #86868B ← 辅助文字
```

**字体**：`font-family="Inter,-apple-system,BlinkMacSystemFont,sans-serif"`

**圆角**：`rx="8"` ~ `rx="12"`（根据区块大小调整）

**线宽**：`stroke-width="1"` ~ `"2"`

#### 三、阴影与深度（必须掌握）

**卡片阴影 — 首选 filter 方式：**

```svg
<defs>
  <filter id="softShadow" x="-15%" y="-15%" width="140%" height="140%">
    <feGaussianBlur in="SourceAlpha" stdDeviation="12"/>
    <feOffset dx="0" dy="6" result="offsetBlur"/>
    <feFlood flood-color="#000000" flood-opacity="0.15" result="shadowColor"/>
    <feComposite in="shadowColor" in2="offsetBlur" operator="in" result="shadow"/>
    <feMerge>
      <feMergeNode in="shadow"/>
      <feMergeNode in="SourceGraphic"/>
    </feMerge>
  </filter>
</defs>
<!-- 使用方式 -->
<rect x="60" y="60" width="400" height="240" rx="12" fill="#FFFFFF" filter="url(#softShadow)"/>
```

**推荐参数：**
- `stdDeviation: 10–16`（越大越柔和）
- `flood-opacity: 0.12–0.20`（太低看不见，太高太重）
- `dy: 4–8`（垂直偏移为主）

#### 六、常见错误与检查清单

| 检查项 | 标准 |
|--------|------|
| ✅ viewBox | 尺寸与内容匹配 |
| ✅ 标签闭合 | 所有标签有开有关 |

### 布局设计师 (Layout Designer)

**唯一职责：将最终分析报告 + SVG 转化为完整的单屏 HTML 文件。**

输入：
- 最终分析报告（来自执笔编辑）
- SVG 代码（图表设计师产出）

输出：`/tmp/url-summary.html`

**执行步骤：**

1. **确认布局**（主编已决策，布局设计师照做）
2. **组装 HTML**：按模板填入内容
3. **嵌入 SVG**：将图表设计师的 SVG 代码放入 `.diagram-wrap` 内
4. **质量自检**：
   - [ ] 单屏不溢出（`overflow: hidden`，100vh）
   - [ ] SVG 标签全部闭合
   - [ ] 无硬编码内容
   - [ ] 叙事内容（非罗列）

    ▼

## 主编审核清单

**执行者：主编（唯一审核者）**

| 检查项 | 标准 | 不通过时打回 |
|--------|------|------------|
| **外部来源** | 有外部信息包，非单一文档来源 | Step 2 |
| **验证完整性** | 主要结论有外部来源支撑 | Step 2 |
| **矛盾处理** | 矛盾信息被如实呈现，不隐藏 | Step 3 |
| **置信度标注** | 结论标注高/中/低置信度 | Step 3 |
| **叙事质量** | 叙事驱动，有观点，有立场 | Step 3 |
| **图表准确性** | SVG 节点数量、连接关系与内容一致；标签闭合 | Step 4 |
| **单屏不溢出** | 100vh 内无滚动条 | Step 6 |
| **视觉规范** | Anthropic 配色、圆角卡片、阴影、hover 动画 | Step 6 |

**审核结论格式：**

```
【主编审核】
审核时间：[年-月-日 HH:MM]

□ 全部通过 → 结论：通过
□ 部分通过（列出未通过项）→ 结论：驳回

未通过项：
1. [检查项] — [具体问题]
2. ...
```

## 驳回与修正机制

### 驳回触发条件

**Step 1（内容分析）被驳回的条件：**
- 未标注盲区
- 纵向/横向分析缺失

**Step 2（外部检索）被驳回的条件：**
- 盲区未被覆盖
- 缺少矛盾信息记录
- 来源可信度低

**Step 3（执笔编辑）被驳回的条件：**
- 未整合外部信息
- 置信度未标注
- 空洞结论未清除
- 出现套话、空洞形容词

**Step 4（图表设计）被驳回的条件：**
- SVG 标签未闭合
- 图表类型与内容分析不匹配
- 节点数量与要点数量不匹配

**Step 6（HTML 生成）被驳回的条件：**
- 100vh 溢出
- 硬编码内容
- 日期不是当天

  --text-primary: #1D1D1F;
  --text-secondary: #6E6E73;

  --radius-lg: 24px;
  --radius-md: 16px;
  --radius-sm: 10px;

**字体**：Inter（Google Fonts），fallback `-apple-system`

**背景**：渐变 `radial-gradient(ellipse at top, #FFFFFF 0%, #F5F5F7 60%, #EBEBED 100%)`

**动画**：卡片 hover 时 `box-shadow` 增强 + `translateY(-2px)`
