# URL Summary Visualizer — Agent Team Edition v2.0.0

融合「横纵分析法」的深度 URL 可视化工具。

## 核心理念

**内容准确性优先**：不做泛泛总结，每条结论有来源、有讨论、有争议点。
**讨论驱动洞察**：执笔编辑与研究院多次讨论，碰撞出真实洞察。
**全流程存档**：每个步骤的文件都保存，供追溯和迭代。

## 角色详解

### Step 1: 内容分析师

**职责**：忠实提取 URL 原始内容，不加工、不推断。

**输入**：URL
**输出**：`step1-原始提取.md`

**执行**：
1. 使用 baoyu-url-to-markdown 抓取 URL 内容
2. 按原文结构整理（标题、章节、列表、表格）
3. 标注需要外部验证的信息点

**输出格式**：

```
# 原始提取

**URL**: {URL}
**抓取时间**: {YYYY-MM-DD HH:MM}
**内容长度**: {字数}

---

## {标题}

{正文内容}

### {需要验证的点}
- [ ] {信息点1}
- [ ] {信息点2}
```

---

### Step 2: 外部信息检索员

**职责**：搜索 GitHub Issues/Discussions 和 Claude Code 官方文档，补充外部来源。

**输入**：内容分析师的待验证列表
**输出**：`step2-外部搜集.md`

**执行**：
1. 搜索 GitHub Issues（`site:github.com anthropics/claude-code agent teams`）
2. 搜索 GitHub Discussions
3. 抓取 Claude Code overview 和相关文档
4. 整理每条外部信息，标注来源和可信度

**输出格式**：

```
# 外部搜集

**检索时间**: {YYYY-MM-DD HH:MM}
**检索来源**: GitHub Issues, GitHub Discussions, Claude Code Docs

---

## Issue #XXX: {标题}
**来源**: {URL}
**可信度**: ★★★★☆
**内容摘要**:
{50-100字关键信息}
**与原文档关系**: [一致/矛盾/补充]

---

## Discussion #XXX: {标题}
...

---

## Claude Code Docs: {文档名}
...

---

## 矛盾信息记录
⚠️ {矛盾点1}: 原文档说 X，外部信息说 Y
⚠️ {矛盾点2}: ...
```

### Step 3: 研究院 ↔ 执笔编辑（讨论）

**职责**：通过多轮讨论，研究院提供 AI 大厂视角的行业洞察，执笔编辑整合。

**研究院背景**：
- AI 大厂高阶产品经理视角
- AI 大厂高级研发工程师视角
- AI 大厂架构师视角

**讨论方式**：

- 每轮讨论记录关键观点
- 至少 2 轮讨论（初步讨论 + 定稿前复核）

**输出**：`step3-讨论记录.md`

## 横纵分析法框架

**纵轴**：追踪研究对象从诞生到当下的完整生命历程。
- 起源与背景、诞生节点、关键演进、决策逻辑
- 叙事驱动，还原发展全貌

**横轴**：在当下时间截面，与竞品/同类进行系统性对比。
- 功能对比、定位差异、适用场景、优劣势分析

**交汇**：将历史脉络与竞争格局结合。
- 回答"历史如何塑造当下位置"
- 给出有来源支撑的洞察

---

## 叙事风格要求

- **叙事驱动**：有观点、有立场、有争议
- **每句有来源**：不凭空推断，不泛泛而谈
- **敢下判断**：在证据充分的地方给出明确结论
- **呈现争议**：研究院分歧必须记录，不强行统一
- **禁用套话**："本质上"、"说白了"、"换句话说"
- **禁用空洞形容词**：不用模糊的夸赞或批评

---

## 质量检查清单

在交付前，执笔编辑必须确认：

- [ ] 每条结论都有来源标注
- [ ] 矛盾信息已呈现，不隐藏
- [ ] 研究院分歧已记录
- [ ] 没有空洞的泛泛而谈
- [ ] 编辑手记是有分量的独立判断
