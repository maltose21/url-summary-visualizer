# URL Summary Visualizer — Agent Team Edition v2.2.0

融合「横纵分析法」的深度 URL 可视化工具。

## 核心理念

**内容准确性优先**：不做泛泛总结，每条结论有来源、有讨论、有争议点。
**讨论驱动洞察**：执笔编辑与研究院多次讨论，碰撞出真实洞察。
**全流程存档**：每个步骤的文件都保存，供追溯和迭代。
**布局设计规范**：参考专业 PPT 设计体系，确保视觉呈现专业、紧凑、无冗余留白。

## 角色详解

### Step 1: 内容分析师

**职责**：忠实提取 URL 原始内容，不加工、不推断。

**输入**：URL
**输出**：`step1-原始提取.md`

**执行**：
1. 使用 baoyu-url-to-markdown 抓取 URL 内容
2. 按原文结构整理（标题、章节、列表、表格）
3. 标注需要外部验证的信息点

**输出格式**：

```
# 原始提取

**URL**: {URL}
**抓取时间**: {YYYY-MM-DD HH:MM}
**内容长度**: {字数}

---

## {标题}

{正文内容}

### {需要验证的点}
- [ ] {信息点1}
- [ ] {信息点2}
```

---

### Step 2: 外部信息检索员

**职责**：搜索 GitHub Issues/Discussions 和 Claude Code 官方文档，补充外部来源。

**输入**：内容分析师的待验证列表
**输出**：`step2-外部搜集.md`

**执行**：
1. 搜索 GitHub Issues（`site:github.com anthropics/claude-code agent teams`）
2. 搜索 GitHub Discussions
3. 抓取 Claude Code overview 和相关文档
4. 整理每条外部信息，标注来源和可信度

**输出格式**：

```
# 外部搜集

**检索时间**: {YYYY-MM-DD HH:MM}
**检索来源**: GitHub Issues, GitHub Discussions, Claude Code Docs

---

## Issue #XXX: {标题}
**来源**: {URL}
**可信度**: ★★★★☆
**内容摘要**:
{50-100字关键信息}
**与原文档关系**: [一致/矛盾/补充]

---

## Discussion #XXX: {标题}
...

---

## Claude Code Docs: {文档名}
...

---

## 矛盾信息记录
⚠️ {矛盾点1}: 原文档说 X，外部信息说 Y
⚠️ {矛盾点2}: ...
```

### Step 8: 布局设计师

**职责**：将终稿 + SVG 整合为 HTML，根据用户选择的屏幕数量适配布局。

**输出**：`/tmp/url-summary-{timestamp}.html`

**屏幕适配规则**：

| 选择 | 布局方式 | 适用场景 |
|------|---------|---------|
| **A) 多屏报告** | 每个分析模块（纵向分析/横向分析/交汇洞察/关键发现/未来推演）独占一屏 | 深度阅读、报告交付 |
| **B) 单屏报告** | 所有内容压缩到 100vh，使用紧凑布局 | 快速概览、高频分享、嵌入展示 |

---

### 单屏报告布局规范（100vh）

当用户选择 B) 单屏报告时，布局必须满足以下约束：

**页面结构**：
- 固定高度：`100vh`，`overflow: hidden`
- 无滚动条（内容自动适配）

**字体压缩规则**：

| 元素 | 多屏基准 | 单屏压缩后 |
|------|---------|-----------|
| 页面主标题 | 36px | 20px |
| 章节标题 | 22px | 14px |
| 内容标题 | 18px | 12px |
| **正文内容** | **16px** | **11px** |
| 注解说明 | 14px | 10px |
| 来源标注 | 12px | 9px |
| 页脚信息 | 13px | 10px |

**布局压缩规则**：

| 元素 | 多屏基准 | 单屏压缩后 |
|------|---------|-----------|
| 页面边距 | 48px | 16px |
| 卡片间距 | 20px | 8px |
| 卡片内边距 | 24px | 12px |
| 行间距 | 1.6em | 1.3em |
| 卡片圆角 | 12px | 6px |

**内容裁剪规则**（单屏必须裁剪，否则溢出）：

1. **来源标注**：合并到一行，用"·"分隔
2. **时间线**：压缩为单行，每个节点只保留标签
3. **表格**：减少列数，合并单元格
4. **关键发现**：只保留标题 + 一句话，删除详细分析
5. **未来推演**：只保留剧本名称 + 概率，删除条件分析
6. **争议点**：完全删除
7. **段落文字**：只保留核心句，删除展开说明

**单屏布局模板**：

```
┌─────────────────────────────────────────────────┐
│ Header (title + meta)                    h: 8%  │

│                                                 │

│  └─────────┘ └─────────┘ └─────────┘ └─────────┘│
│                                                 │
├─────────────────────────────────────────────────┤
│  ┌──────────────────┐  ┌──────────────────────┐│
│  │  未来推演        │  │  Token成本/竞品对比  ││ h: 30%
```

### 内容密度原则

**每页信息量**：
- 封面/结论页：3-5 个要点
- 分析页：5-8 个要点
- 详情页：8-12 个要点

**避免**：
- 大段留白（超过页面 20%）
- 字号过小（正文字号小于 16px）
- 行距过松（行间距小于 1.4em）
- 内容过密（每屏超过 15 个要点）

### 图表设计规范

**图表尺寸**：
- 宽度：占内容区 60-100%
- 高度：根据内容比例自适应，最小 200px

**图表元素**：
- 标题：18-20px，Bold
- 标签：12-14px，Regular
- 数值：24-32px，Bold（关键数据）
- 图例：12-14px，Regular

**颜色应用**：
- 主数据：`#0066CC`
- 辅助数据：`#86868B`
- 强调数据：`#FF3B30` 或 `#34C759`
- 背景：`#F5F5F7`
- 网格线：`#E5E5E7`

## 横纵分析法框架

**纵轴**：追踪研究对象从诞生到当下的完整生命历程。
- 起源与背景、诞生节点、关键演进、决策逻辑
- 叙事驱动，还原发展全貌

**横轴**：在当下时间截面，与竞品/同类进行系统性对比。
- 功能对比、定位差异、适用场景、优劣势分析

**交汇**：将历史脉络与竞争格局结合。
- 回答"历史如何塑造当下位置"
- 给出有来源支撑的洞察

---

## 叙事风格要求

- **叙事驱动**：有观点、有立场、有争议
- **每句有来源**：不凭空推断，不泛泛而谈
- **敢下判断**：在证据充分的地方给出明确结论
- **呈现争议**：研究院分歧必须记录，不强行统一
- **内容充实**：每个分析点要有足够的深度和细节，不空洞
- **禁用套话**："本质上"、"说白了"、"换句话说"
- **禁用空洞形容词**：不用模糊的夸赞或批评
